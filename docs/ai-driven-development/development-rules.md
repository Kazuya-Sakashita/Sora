# AI駆動開発ルール — Sora

## 開発サイクル

```
評価（現状把握）
    ↓
ISSUE化（問題を個別に切り出す）
    ↓
優先順位付け（Critical → High → Medium → Low）
    ↓
1件ずつ実装
    ↓
動作検証（ブラウザ確認＋既存機能チェック）
    ↓
ISSUEクローズ
    ↓
次のISSUEへ
```

このサイクルを回し続ける。まとめてやらない、飛ばさない。

---

## 評価のやり方

### アプリ評価を依頼するときのプロンプト

```
以下の観点でSoraアプリを厳しく評価してください。

【評価観点】
- ユーザーが最初に開いたとき、何のアプリかわかるか
- 生前利用として自然に入れるか（ペットロス専用に見えないか）
- 記録の導線が直感的か
- 感情設計が適切か（重すぎ/軽すぎ/冷たくないか）
- MVPとして必要な機能が揃っているか
- データ永続化が機能しているか
- 既存の実装で明らかにおかしい点

【出力形式】
各問題を以下のフォーマットで列挙してください：
- 優先度（critical/high/medium/low）
- 問題の概要（1行）
- 詳細（2〜3行）
- 影響する画面・コンポーネント
```

## ISSUE管理ルール

### ISSUEファイルの作成場所

```
docs/issues/ISSUE-001-title.md
docs/issues/ISSUE-002-title.md
...
```

番号は連番。タイトルはケバブケース（英語またはローマ字）。

### ISSUEファイルのテンプレート

`docs/ai-driven-development/issue-template.md` を参照。

### ステータス管理

| ステータス | 意味 |
|-----------|------|
| `open` | 未着手 |
| `in_progress` | 対応中 |
| `done` | 完了・動作確認済み |
| `wont_fix` | 対応しないと決定（理由を書く） |

## Claude Codeへの指示テンプレート

### 実装指示（標準）

```
ISSUE-XXX を対応してください。

【対象ISSUE】docs/issues/ISSUE-XXX-title.md を確認してください
【実装前】影響する画面・コンポーネントを特定して教えてください
【実装後】変更した画面とその他の画面が正常に動くか確認してください
【完了後】ISSUEのステータスを done に更新してください
```

### 評価指示

```
現在のSoraアプリ（pnpm dev で起動中）を以下の観点で評価し、
問題点をISSUEとして docs/issues/ に作成してください。

観点：[具体的な観点を書く]

優先度はcritical/high/medium/lowで判断し、
criticalから順に番号を振ってください。
```

### 影響範囲確認指示

```
ISSUE-XXX の実装前に、変更が必要なファイルと
その変更が影響しうる他の画面・コンポーネントを確認して教えてください。
実装はまだしないでください。
```

---

## 優先度の判断基準

| 優先度 | 基準 |
|--------|------|
| **critical** | アプリが機能しない・データが消える・クラッシュする |
| **high** | MVPの中心機能が欠けている・UXが大きく損なわれる |
| **medium** | あると良い改善・感情設計の改善・細かいUX問題 |
| **low** | 将来対応でよい・Phase 2以降の機能 |

## 動作検証チェックリスト

実装後、以下を確認してから done にする。

### 変更画面の確認
- [ ] 意図した変更が反映されているか
- [ ] エラーが出ていないか（コンソール確認）
- [ ] スマートフォンサイズ（390px幅）で見た目がおかしくないか

### 既存機能の確認（デグレチェック）
- [ ] オンボーディング → プロフィール作成 → ホームの導線が動くか
- [ ] 変更した画面以外のナビゲーションが正常か
- [ ] データが意図どおりに保持されているか
