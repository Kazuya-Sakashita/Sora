# アプリ評価フレームワーク

## 評価タイミング

| タイミング | 評価の深さ |
|-----------|-----------|
| 機能実装直後 | 変更箇所のみ簡易確認 |
| 3〜5件ISSUE完了後 | HEART評価 |
| MVP完成時 | HEART + AARRR + 技術評価 |

---

## HEART評価

Googleが開発したUX評価フレームワーク。感情・継続・機能を測る。

| 指標 | 意味 | Soraでの確認観点 |
|------|------|----------------|
| **H**appiness | 満足度・感情的な質 | 触ったときに「ほっとする」か。コピーが温かいか |
| **E**ngagement | エンゲージメント | 記録を続けたくなるか。アプリを開く動機があるか |
| **A**doption | 採用 | 初回から主要機能を使えているか |
| **R**etention | 継続 | 翌日・翌週もアプリを開きたいと思うか |
| **T**ask Success | タスク完了 | 記録追加・予定追加が3タップ以内でできるか |

### 評価方法

各指標を 1〜5 で採点し、理由を1〜2行で書く。

```markdown
## HEART評価（YYYY-MM-DD）

| 指標 | スコア | 理由 |
|------|--------|------|
| Happiness | 4 | 色・余白は温かいが、コピーがまだ重い箇所あり |
| Engagement | 2 | 記録後の達成感がない。続けたくなる動機が弱い |
| Adoption | 3 | オンボーディングはシンプルだが導線が1箇所不明 |
| Retention | 2 | 「また開きたい」理由が現状では弱い |
| Task Success | 4 | 記録追加は3タップで完了できている |
```

## AARRR評価

成長・事業視点での評価。MVPでは参考程度でよい。

| 指標 | Soraでの確認観点 |
|------|----------------|
| **A**cquisition | App Storeのスクショ・説明文が「生前から使う」と伝わるか |
| **A**ctivation | 初回起動〜最初の記録追加が5分以内にできるか |
| **R**etention | 1週間後もアプリを開いているユーザーが想像できるか |
| **R**eferral | 「友人に勧めたい」と思えるか。何を軸に勧めるか |
| **R**evenue | プレミアム機能に課金したいと思える体験があるか |

---

## 技術評価

| 観点 | 確認内容 |
|------|---------|
| パフォーマンス | 画面遷移が500ms以内か |
| エラーハンドリング | コンソールにエラーがないか |
| 型安全性 | TypeScriptの型エラーがないか |
| データ整合性 | localStorage のデータが意図通りに読み書きされているか |
| レスポンシブ | 390px（iPhone）で崩れていないか |

## gstackを使った評価手順

### 準備

```bash
pnpm dev  # サーバー起動
# gstack でブラウザを開き、スクリーンショットを撮りながら確認
```

### 確認シナリオ（優先順）

1. **初回起動** — オンボーディング全画面をスクリーンショット
2. **ペット登録** — 必須項目だけで完了できるか
3. **最初の記録追加** — 写真+タグ1つで3タップ以内か
4. **タイムライン確認** — 1件でも美しく見えるか
5. **予定追加** — カテゴリ選択〜完了までの流れ

---

## 評価 → ISSUEへの落とし込み方

1. 評価結果を書き出す（上記フレームワークに沿って）
2. スコアが3以下の観点から問題を抽出する
3. 各問題を1件のISSUEとして `docs/issues/` に作成する
4. 優先度を付けて `docs/issues/README.md` に追記する

```markdown
## 評価からISSUEへの変換例

評価: Engagement = 2「記録後の達成感がない」
↓
ISSUE-006: 思い出記録後の完了フィードバックがない（high）
```
