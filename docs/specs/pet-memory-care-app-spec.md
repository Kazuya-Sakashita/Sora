## 1. 企画概要

### 1.1 アプリの本質

このアプリは「ペット管理ツール」でも「グリーフケアアプリ」でもない。

**「ペットとの日常の愛情を記録し、それが自然に未来の支えになる体験」** を提供するアプリである。

ペットと暮らしている間、人は無意識に無数の愛情を積み重ねている。トリミングの予定を入れる。病院に連れて行く。一緒に旅行する。特別でない何でもない日の写真を撮る。それらはすべて愛情の痕跡だ。

このアプリはその痕跡を、丁寧に、美しく、自然に積み重ねていく。そして──ペットとの別れが訪れたとき、ユーザーが蓄積してきたすべての記録が、「あなたはこれだけ愛していた」という証として、静かに寄り添う。

特別な機能を「亡くなった後のために」用意するのではない。生きている間の普通の日々の記録が、そのまま癒しの素材になる。これが、他のすべてのアプリとの本質的な違いである。

### 1.2 アプリ名候補

感情的に選ばれ、心に残る名前を最優先とする。

| # | アプリ名 | 読み・ニュアンス |
|---|----------|----------------|
| 1 | **Sora** | 空・自由・ペットが帰る場所・日本語として美しい |
| 2 | **Hana** | 花・生命・咲いていた時間への敬意 |
| 3 | **Kizuna** | 絆・つながり・生前も死後も続く愛情 |
| 4 | **Niji** | 虹・虹の橋・希望・やさしさ |
| 5 | **Nukumori** | ぬくもり・体温・一緒にいた記憶の感触 |
| 6 | **Akari** | 灯り・ペットが照らしてくれたもの |
| 7 | **Hoshi** | 星・夜空にいる存在・永遠 |
| 8 | **Tomo** | 友・いつも一緒にいた存在 |
| 9 | **Irodori** | 彩り・ペットがくれた日々の色 |
| 10 | **Shizuku** | 雫・涙と愛情・静かな感情の器 |

**推奨:** `Sora`（空）。シンプルで国際展開も視野に入る。空は「境界がなく、ペットがいた場所にも、いる場所にもなれる」概念として機能する。

### 1.3 タグライン案

| # | タグライン |
|---|-----------|
| 1 | **「そばにいた時間が、ずっとそばにいてくれる。」** |
| 2 | **「愛してた日々を、ずっと愛せる場所。」** |
| 3 | **「記録じゃなく、愛情を残す。」** |
| 4 | **「いつかの自分を、今日の記録が支える。」** |
| 5 | **「一緒にいる今が、ずっと続く形になる。」** |

**推奨:** タグライン1「そばにいた時間が、ずっとそばにいてくれる。」  
過去形と現在進行形が共存する文構造が、生前〜ロス後をシームレスにつなぐ世界観を言語化している。

## 2. 課題設定

### 2.1 ペットロスの現実

日本では年間推計60〜80万頭以上の犬猫が死亡している（ペットフード協会推計）。ペットロスは精神的に深刻な影響を与え、うつ状態、不眠、食欲不振、社会的孤立を引き起こすことがある。にもかかわらず、日本社会でペットロスへの理解は十分でなく、「たかがペット」という空気が当事者を孤立させる。

### 2.2 既存手段の限界

| 既存手段 | 課題 |
|----------|------|
| ペット管理アプリ（PetNote等） | 記録・スケジュール管理が目的。感情的価値が薄い。ロス後に開けない |
| SNS（Instagram等） | 記録はできるが体系化されない。見返す設計ではない |
| 日記アプリ | テキスト中心。写真・予定との統合がない。ロス後体験の設計なし |
| グリーフサポートサービス | ロス後専門で、生前から使えない。ロス体験を前提とした設計が重い |
| YouTube・コンテンツ | 受動的消費。個人の記憶とつながらない |

### 2.3 解決すべき真の問題

**問題A（生前）:** 愛情を注いでいる日々を、後から振り返れる形で残せていない。日常の記録は断片化し、感情と切り離されている。

**問題B（喪失直後）:** 急に「記録しよう」「振り返ろう」とはなれない。何かに寄り添いたいが、重すぎるコンテンツには耐えられない。

**問題C（回復期）:** 悲しみが薄れることへの罪悪感がある。思い出を「過去のもの」にしたくない。しかし執着することへの不安もある。

**問題D（社会的）:** ペットロスを話せる場が少ない。孤独に抱えているユーザーが多い。

## 3. プロダクトコンセプト

### 3.1 コンセプト定義

> **「愛情の記録が、未来の支えになる。生きている今から、ずっと一緒に。」**

このアプリの設計思想は「グリーフケアは、別れの後に始まらない」である。

ペットと過ごす毎日の記録──何気ない写真、病院の記録、トリミングの日、笑えるエピソード──それらが積み重なるほど、別れた後に「あなたはこれだけ愛していた」という証が育まれる。

ロス後専用の機能を無理に用意するのではなく、生前の自然な利用がそのままロス後の癒しになる設計を徹底する。

### 3.2 類似アプリとの本質的な違い

| 観点 | 一般ペット管理アプリ | グリーフアプリ | **このアプリ** |
|------|---------------------|--------------|---------------|
| 利用開始タイミング | 生前 | 亡くなった後 | **生前から自然に始まる** |
| 主要価値 | 管理の効率化 | 悲しみのケア | **愛情の積み重ねと自然な寄り添い** |
| 継続動機 | 利便性 | 必要性 | **「残したい」という感情欲求** |
| ロス後体験 | 見返しにくい | 専用設計 | **生前の記録が自然に支えになる** |
| 感情設計 | 薄い | 重すぎる | **日常の中に静かに宿る** |
| AI活用 | なし・補助的 | チャット中心 | **記録を読み取り、寄り添いの言葉を生む** |

### 3.3 なぜ「生前から使う設計」が重要か

**理由1: 記録の密度が癒しの質を決める**  
ロス後に「もっと写真を撮っておけばよかった」「日々を記録しておけばよかった」という後悔は普遍的だ。生前から使うことで、この後悔を事前に解消する。記録が豊富であるほど、ロス後の振り返り体験の質が上がる。

**理由2: 重い目的として入らないための設計**  
「ペットロスに備えるアプリ」として提示すると、誰も使わない。「ペットとの日々をもっと豊かにするアプリ」として提示し、自然な利用の延長線上にロスケアが来る設計が不可欠だ。

**理由3: データが感情コンテキストを持つ**  
生前から蓄積された記録には、飼い主の感情・関係性・日常の質感が埋め込まれている。ロス後にAIが寄り添うとき、このコンテキストがあることで「あなたとリクの話」ができる。汎用的なグリーフトークではなく、その人だけの体験になる。

**理由4: 別れへの心理的準備になる**  
高齢や病気のペットを持つ飼い主が記録を積み重ねる行為は、愛情を確認し続ける行為でもある。これが、別れが来たとき「後悔なく愛していた」という感覚につながる。

## 4. ターゲット

### 4.1 メインターゲット

**犬・猫を飼っている25〜45歳の日本在住女性**  
（男性も対象だが、感情記録・共有行動は女性が先行する傾向）

- ペットを「家族」として捉えている
- SNSで日常を発信しているが、体系的に残せていないことに微妙な不満がある
- 将来の別れを漠然と恐れているが、具体的に何かをしているわけではない
- 感情的価値に課金する意識がある（スタバ・サブスク型サービスへの抵抗が薄い）

### 4.2 サブターゲット

**A: ペットロスを経験した直後（30〜60代）**  
- 別れから数日〜数週間以内
- 何かに寄り添いたいが、重いコンテンツには耐えられない
- 過去の記録を残したい・見返したいが、道具がない

**B: 高齢・闘病ペットの飼い主（35〜65代）**  
- 「あと何年一緒にいられるか」を意識し始めている
- 記録を残しておきたい気持ちが高まっている
- 具体的なケアツールを探している

**C: ペットを迎えたばかりの若い飼い主（20〜35歳）**  
- 最初から全部記録したい
- 成長を残したい
- 感情価値より体験価値に重きを置く

### 4.3 ターゲット外（意図的に外すユーザー）

- ペットを管理対象として捉えているユーザー
- グリーフカウンセリングの代替として期待するユーザー
- 記録習慣がまったくないユーザーで負荷を嫌うユーザー

### 4.4 フェーズ別価値提供

| フェーズ | ユーザー状態 | アプリが提供する価値 |
|----------|------------|-------------------|
| 一緒に暮らしている時 | 幸せ・充実・日常 | 「記録する楽しさ」「振り返る喜び」 |
| 高齢・病気で不安 | 焦り・悲しみの予感・愛情の高まり | 「今を残す大切さ」「記録による愛情の確認」 |
| 亡くなった直後 | 混乱・悲しみ・空白感 | 「積み上げた愛情の証」「静かな存在感」 |
| 少し時間が経った後 | 悲しみ+懐かしさ+前向きな気持ち | 「思い出と共に生きる感覚」「新たな関係への移行」 |

## 5. 提供価値

### 5.1 機能価値

- ペットの日常・健康・予定をひとつにまとめて管理できる
- 写真・メモ・感情記録を時系列で残せる
- AIが記録をもとに、やさしい言葉で振り返りを届ける

### 5.2 感情価値

- 「残してよかった」という充足感
- 「こんなに愛していた」という自己確認
- ロス後に「ひとりじゃない」という安心感
- 記録が癒しになる体験

### 5.3 社会価値

- ペットロスを一人で抱えない体験の提供
- ペットとの愛情を可視化することへの社会的理解の醸成
- 「動物と人の絆」を体験として次世代に残す

## 6. ユーザーストーリー

### 6.1 生前フェーズ

```
As a 日常的なペットの飼い主,
I want to ペットとの日々を美しく残したい,
So that 振り返ったときに「本当によい時間だった」と感じられる.
```

```
As a 高齢ペットを持つ飼い主,
I want to 今のうちにできるだけ多くの記録を残したい,

```

```
As a 病院やトリミングの予定を管理したい飼い主,
I want to スケジュール管理と感情記録が同じ場所にある,
So that 日常と感情が切り離されずに積み重なる.
```

### 6.2 ロス後フェーズ

```
As a ペットを亡くしたばかりの飼い主,
I want to 過去の記録を静かに振り返りたい,
So that 「あの子はここにいた」という感覚を持ち続けられる.
```

```
As a ペットロスから少し回復してきた飼い主,
I want to 悲しみだけでなく、楽しかった記憶に触れたい,
So that 思い出を「過去のもの」にせず、「今も続くもの」として感じられる.
```

```
As a ペットロスを経験した飼い主,
I want to AIとの会話で自分の気持ちを整理したい,
So that 孤独感が和らぎ、自分のペースで前に進める.
```

## 7. 機能一覧

### 7.1 必須機能（MVP）

#### 7.1.1 ペットプロフィール管理

**なぜ必要か:**  
すべての記録の「主語」となる存在を登録することで、アプリ全体に人格と感情が宿る。「あなたのペット」ではなく「リクのページ」になることで、愛着が生まれ継続率が上がる。

| 項目 | 内容 |
|------|------|
| 登録内容 | 名前、種類、品種、誕生日、性別、プロフィール写真 |
| 任意項目 | ニックネーム、好きなもの・苦手なもの、特徴メモ |
| 複数ペット | 複数頭登録対応（タブ切り替えUI） |
| ステータス | 「今一緒にいる」「虹の橋を渡った」の選択（ロス後の切り替えポイント） |

#### 7.1.2 思い出記録（フォト＋メモ）

**なぜ必要か:**  
このアプリの核心。写真だけ・テキストだけでなく、両者と「その日の気持ち」が一緒に残ることで、記録が感情の痕跡になる。後から見たとき、その日の空気感まで蘇る設計を目指す。

| 項目 | 内容 |
|------|------|
| 入力形式 | 写真（複数可）＋ テキストメモ ＋ 気持ちタグ |
| 気持ちタグ | 「うれしい」「おだやか」「心配」「笑った」「愛おしい」など感情ベース |
| タイムスタンプ | 自動（撮影日時取得）or 手動設定 |
| カテゴリ | 「旅行」「日常」「病院」「トリミング」「記念日」「その他」 |
| タグ | 自由入力タグ |

#### 7.1.3 予定管理（カレンダー統合）

**なぜ必要か:**  
スケジュール管理のために使い始めたユーザーが、感情記録まで自然に行動を拡張できる。「病院の予定」→「通院後の様子メモ」という導線が生まれる。

| 項目 | 内容 |
|------|------|
| 予定種別 | 通院、トリミング、ワクチン、フィラリア予防、記念日、その他 |
| 繰り返し | 月次・年次の繰り返し設定 |
| リマインダー | 前日・当日の通知 |
| 通院後記録 | 予定完了時に「記録を残す」への自然な誘導 |

#### 7.1.4 思い出タイムライン

**なぜ必要か:**  
記録の積み重ねを「見られる形」にすることで、アプリを開く動機が生まれる。タイムラインは「時間の川」であり、そこを流れる記録がペットとの歴史になる。

| 項目 | 内容 |
|------|------|
| 表示形式 | 時系列（逆順）スクロール、フォト主体カード型 |
| フィルタ | カテゴリ別、年別、月別、気持ちタグ別 |
| 振り返り演出 | 「1年前の今日」表示（日常的な感情トリガー） |
| ハイライト | 自動または手動でのベスト記録ピックアップ |

#### 7.1.5 気持ち記録（ムードログ）

**なぜ必要か:**  
ペットの状態だけでなく、飼い主自身の感情の流れを残すことで、ロス後に「あの頃どんな気持ちだったか」を確認できる。これがAIの寄り添いの文脈になる。

| 項目 | 内容 |
|------|------|
| 入力 | 今日の気持ち（絵文字 or タグ選択）＋ ひと言メモ（任意） |
| 頻度 | 毎日の習慣として設計（通知は任意） |
| 連動 | その日の記録と自動紐付け |

### 7.2 差別化機能（MVP後期〜Phase 2）

#### 7.2.1 AI振り返り・寄り添い会話

**なぜ必要か:**  
記録を「読んでくれる存在」がいることで、記録行為に意味が増す。ロス後には、AIがユーザーの記録を踏まえて「あなたとリクの話」ができる。汎用AIではなく、記録に基づいたパーソナルな対話が感情価値を生む。

| 項目 | 内容 |
|------|------|
| 生前利用 | 最近の記録から「やさしい振り返りメッセージ」を生成（週1回程度） |
| 記念日・特別な日 | 「今日はリクのトリミングから1年ですね」などの自然な語りかけ |
| ロス後利用 | 過去の記録をもとにした寄り添い会話（非チャットボット的な設計） |
| 生成内容の質 | 感情的に安全な言葉選び・押し付けなし・問いかけベース |
| 距離感 | 「答えを出す」のではなく「一緒にいる感覚」を重視 |

**AIの設計原則:**
- 診断・アドバイスをしない
- 「きっとこうだった」という断定をしない
- 「あなたの気持ちは正しい」という肯定過多を避ける
- ユーザーが語りたいときに語れる空間を提供する

#### 7.2.2 思い出から手紙・言葉を生成する機能

**なぜ必要か:**  
ロス後、言葉を整理できない状態のユーザーに対して、AIが記録から「あなたが伝えたかったこと」を代わりに言語化する機能。「あなたがリクに伝えたかった言葉」として生成することで、追悼・感謝の感情が結晶化する。

| 項目 | 内容 |
|------|------|
| 入力 | 記録データ（写真・メモ・気持ちログ）＋ ユーザーの気持ちひと言 |
| 出力 | 「あなたからリクへの手紙」「リクとの日々を振り返る言葉」 |
| 形式 | テキスト（保存・シェア可能）、将来的にはカード形式 |
| 注意設計 | 生成前に「今の気持ちで大丈夫ですか？」の確認UI |

#### 7.2.3 「今日の記憶」プッシュ通知

**なぜ必要か:**  
ユーザーが意識していないタイミングで記録に触れさせる。「1年前の今日」通知は、感情的なトリガーとして非常に効果が高い。SNS的な「過去の記憶」機能のペット版。

| 項目 | 内容 |
|------|------|
| 通知内容 | 「n年前の今日のリク」 ＋ 写真サムネイル |
| 頻度 | 記録がある日のみ（毎日は出さない） |
| 設定 | ON/OFF、時間帯設定 |
| ロス後 | ロス後も継続（感情的な配慮として、最初の1ヶ月はデフォルトOFF） |

### 7.3 将来機能（Phase 3以降）

- 家族共有機能（複数アカウントで同じペットの記録を共有）
- 思い出アルバム自動生成（年鑑・フォトブック出力）
- 健康記録・体重トラッキング
- ペットロスコミュニティ（匿名・オプトイン）
- 思い出のデジタル遺影・メモリアルページ

## 8. フェーズ設計

### 8.1 日常フェーズ

**ユーザー心理:**  
幸せで充実している。ペットは元気で、毎日が当たり前に続くと感じている。記録は「楽しいから」「残したいから」という動機で行う。この段階でアプリへの習慣が形成される。

**アプリが提供すべき価値:**
- 記録の楽しさ・達成感
- 「残している」という充足感
- 振り返りの喜び

**出してよい機能:**
- 全機能（記録・予定・タイムライン・AI振り返り）
- 「今日の記憶」通知
- カレンダービュー

**出しすぎると逆効果な機能:**
- 「老後の備え」を想起させるコピー
- 健康指標の過度な可視化（不安を煽る）
- ロス後を匂わせるUI要素

---

### 8.2 体調不安フェーズ

**ユーザー心理:**  
老齢や病気により、「いつかは別れが来る」という現実が重くのしかかっている。焦りと悲しみの予感の中で、「今できることをしておきたい」という衝動が高まる。記録行動が増える傾向がある。

**アプリが提供すべき価値:**
- 「今残しておいて良かった」という安心感
- 通院記録・投薬記録など実務サポート
- 感情的な「今日も一緒にいた」という記録の積み重ね

**出してよい機能:**
- 通院・投薬記録の強化
- 気持ち記録（飼い主自身の感情記録）
- AI振り返りメッセージ（穏やかな内容）

**出しすぎると逆効果な機能:**
- 「いつかのために記録を」という直接的な促し
- ペットロス後のコンテンツへの誘導
- 健康指標グラフの強調表示（悪化が見えるため）

---

### 8.3 別れ直後フェーズ

**ユーザー心理:**  
混乱・空白・痛み。何もする気になれないが、ペットに関係するものに触れたい気持ちもある。新しいことを始める余裕はなく、「今あるものを」「静かに」「強制なく」触れたい。

**アプリが提供すべき価値:**
- 積み上げた記録に静かにアクセスできること
- 「ここにいた」という存在感の確認
- 強制されない、でも逃げ場になる体験

**出してよい機能:**
- タイムライン閲覧（写真・記録の振り返り）
- 気持ち記録（「今日の気持ち」をひと言残す程度）
- 静かな通知停止（デフォルトでリマインダー停止）

**出しすぎると逆効果な機能:**
- AI会話への積極的な誘導（タイミングが早すぎる）
- 「次のペットを...」への言及
- 「前を向いて」系のコピー
- 通知・バッジの多発
- 新機能の案内・アップセル

**設計上の重要ポイント:**  
ペットのステータスを「虹の橋を渡った」に切り替えた後、アプリの雰囲気が静かに変わる演出を検討する。強制ではなく、ユーザーの選択で切り替える。切り替え後も、今まで積み上げた記録はすべてそのままアクセスできる。

---

### 8.4 回復フェーズ

**ユーザー心理:**  
悲しみは続いているが、少しずつ「懐かしい」「また思い出を見たい」という気持ちが芽生える。罪悪感（楽しい気持ちになることへの後ろめたさ）と、前向きな感情の間で揺れる。

**アプリが提供すべき価値:**
- 思い出を「楽しく」「自分のペースで」振り返れること
- AIとの対話による感情の整理
- 「前を向くこと」を押し付けず、「ペットとの記憶と共に歩く」感覚

**出してよい機能:**
- AI寄り添い会話
- 思い出から手紙・言葉を生成する機能
- 「今日の記憶」通知（ユーザーが選択できる）
- 思い出アルバム

**出しすぎると逆効果な機能:**
- 「完全に乗り越えましょう」的コピー
- 新しいペットへの誘導（ユーザーから言わない限り触れない）

## 9. MVP定義

### 9.1 MVP最小構成

MVPで作るのは「生前から使えて、ロス後にも自然につながる、最小限の美しいアプリ」である。機能の多さではなく、体験の質を最優先とする。

**MVP必須機能（この5つだけで始める）:**

1. **ペットプロフィール登録** — 名前・写真・誕生日・種類
2. **思い出記録** — 写真＋メモ＋気持ちタグ（シンプルな3タグ：うれしい/おだやか/心配）
3. **予定管理** — 通院・トリミング・ワクチン・記念日の4種類のみ
4. **タイムライン** — 記録を時系列で見返す
5. **「1年前の今日」通知** — 感情的エンゲージメントの起爆剤

**MVPに含めない（意図的に削る）もの:**

| 機能 | 削る理由 |
|------|---------|
| AI会話 | コスト・品質担保・感情設計の難易度が高い。記録が蓄積されてから意味を持つ |
| 手紙生成 | 同上。最初にあっても使われない |
| 家族共有 | 複雑性が跳ね上がる |
| 健康記録 | ペット管理アプリとの差別化を守るために削る |
| コミュニティ | モデレーション・安全設計のコストが高い |
| ペットステータス切り替え（ロス後UI） | MVP段階では生前ユーザーの体験最適化を優先 |

### 9.2 MVPで検証する仮説

| # | 仮説 | 検証方法 |
|---|------|---------|
| H1 | ユーザーは「感情タグ付きの記録」に魅力を感じ、継続的に記録する | 週次記録率・気持ちタグ利用率 |
| H2 | 予定管理（通院等）が記録習慣の入口になる | 予定登録→記録への遷移率 |
| H3 | 「1年前の今日」通知がアプリへの再帰を促す | 通知からの起動率・記録追加率 |
| H4 | 写真＋感情記録の組み合わせが既存アプリより「残したい」動機を高める | NPS・継続率・口コミ率 |

### 9.3 「手応えあり」の判断基準

- **Week 1継続率:** 60%以上（記録を1件以上追加）
- **Month 1継続率:** 30%以上（週1回以上起動）
- **記録密度:** ユーザーあたり月5件以上の記録
- **NPS:** 50以上（「友人に勧めたい」が強い）
- **定性:** 「このアプリを使っていてよかった」という言語化

### 9.4 広げすぎで危険なポイント

- AI機能を早期に入れると「感情的に不安全な体験」が先に出てしまうリスク
- ロスケア機能を目立たせると「死ぬための準備アプリ」に見えるリスク
- 機能を詰め込むと「ただの多機能ペット管理アプリ」に見えるリスク
- 通知を多くするとペットロス直後のユーザーへの配慮不足になるリスク

## 10. 将来拡張

### Phase 2（MVP検証後、3〜6ヶ月）

**AI振り返り・寄り添い会話**  
蓄積した記録をもとに、週1回のAIからのやさしい振り返りメッセージ。ロス後には記録ベースの対話機能を解放。記録が少ない段階での提供は品質が担保できないため、Phase 1の蓄積後に実装する。

**ペットステータス切り替え（ロス後UI）**  
「虹の橋を渡った」状態への切り替えUI。切り替え後はアプリの雰囲気が静かになる（色調・フォント・コピーの変化）。記録はすべてそのまま。

**思い出から手紙・言葉を生成する機能**  
AI振り返りの延長として、ユーザーの記録から「あなたがペットに伝えたかった言葉」を生成。アプリ内保存・シェア対応。

### Phase 3（6〜12ヶ月）

**家族共有**  
同居の家族が同じペットの記録を共同で残せる。記録の「書いた人」が表示される。家族の視点で記録が豊かになる。

**思い出アルバム自動生成**  
月次・年次でアルバムを自動生成。テーマ別（旅行・日常・病院）のアルバム。将来的にはフォトブック出力。

### Phase 4（12ヶ月以降）

**メモリアルページ（デジタル遺影）**  
ロス後のユーザーが任意で作れる、ペットの「記念ページ」。記録・写真・言葉が集約される。公開設定は任意。

**ペットロスコミュニティ（匿名・招待制）**  
同じ体験をした人と繋がれる匿名スペース。モデレーター必須。感情安全の設計を徹底する。

## 12. UX方針

### 12.1 基本UX原則

**原則1: 記録の摩擦を最小化する**  
写真を選んで気持ちタグを1つ押すだけで記録が完了する。テキストは任意。「完璧に残す」より「気軽に残す」を優先する。

**原則2: 振り返りを驚きとして届ける**  
ユーザーが意識していないタイミングで「1年前の今日」通知を届ける。「残しておいてよかった」という感情が、継続動機になる。

**原則3: 感情に寄り添い、感情を追い立てない**  
AIのメッセージ・コピー・通知すべてにおいて、「行動を促す」のではなく「一緒にいる」感覚を大切にする。

**原則4: 重さと軽さのバランス**  
感情的に重いテーマだが、UIは軽く・触りやすく・開くことへのハードルを低く保つ。色は暖色系・余白多め・フォントは柔らかい書体。

**原則5: 押し付けない**  
通知はオプション。AI機能は任意起動。ロスケア機能はユーザーが選択したときだけ提示する。

### 12.2 デザインシステム方針

| 要素 | 方針 |
|------|------|
| カラー | 温かみのあるニュートラル（クリームホワイト・テラコッタ・セージグリーン・アンバー） |
| フォント | 日本語: 丸みのあるゴシック（Noto Sans JP/UDデジタル教科書体系）, 英語: Lato/DM Sans |
| 余白 | 広め（コンテンツを詰め込まない） |
| アイコン | 線画ベースでやわらかいスタイル |
| 写真 | 全面表示・角丸・シャドウなし（素材を見せる） |
| アニメーション | 穏やかなイーズ（速すぎない・派手すぎない） |
| トーン | 優しい・穏やか・生活感・光を感じる |

### 12.3 コピーライティング原則

- 「管理」「記録する」という言葉より「残す」「大切にする」を優先
- 命令形より問いかけ形（「記録してください」→「今日の思い出、残しませんか？」）
- ロスケア文脈では「乗り越える」「立ち直る」を使わない
- AIの言葉は「です/ます」調で穏やかに。「！」多用禁止
- エラーメッセージも優しいトーンで

## 13. 倫理・リスク整理

### 13.1 センシティブ領域としての設計原則

このアプリはペットロスという、当事者にとって精神的に深刻な体験を扱う。医療・心理ケアの代替を目指さず、補助的・感情的サポートに徹することを設計原則とする。

### 13.2 表現上のNG

| NGカテゴリ | 具体的なNG表現 | 代替方針 |
|-----------|--------------|---------|
| 断定的な死後観 | 「天国で待っている」「虹の橋で会える」 | 「多くの人がそう感じています」という表現に留める |
| 宗教的断定 | 特定の宗教観に基づく表現 | 普遍的な感情・体験として語る |
| 強制的な前向き | 「前を向いて」「新しいペットを」 | 触れない。ユーザーから言わない限り言わない |
| 医療行為の代替 | 「このアプリで癒されます」「心の治療になります」 | 「寄り添う」「そばにいる」レベルにとどめる |
| 依存を促す | 「毎日必ず開いてください」 | 「気が向いたときに」「あなたのペースで」 |

### 13.3 AIの距離感設計

AIは「カウンセラー」でも「友人の代替」でも「ペットの代替」でもない。

**AIが担う役割:**
- 記録を読んで、ユーザーの愛情を「言語化」すること
- ユーザーが語りたいとき、静かに耳を傾けること
- 「あなたはこれだけ愛していた」という事実を確認すること

**AIが担わない役割:**
- 診断・アドバイス・処方
- 「リクがこう思っているはずです」という断定
- ペットの擬似人格化（「リクからのメッセージ」として語らせない）
- 長期的な感情依存の形成

**AIの応答原則:**
- 短く・静かに・問いかけで返す
- 答えを出さない
- ユーザーが「AIに依存しすぎているかも」と感じたとき、自然に「リアルの人間との時間も大切に」と促せる設計を持つ（ただし押しつけない）

### 13.4 医療・心理ケアとの線引き

アプリ内に以下を明示する:

> 「このアプリは、ペットとの思い出を大切にするためのものです。深刻な悲しみや精神的な不調を感じている場合は、専門家のサポートをご検討ください。」

「専門家へのリンク」を設定画面に静かに配置する（目立たせない）。

### 13.5 依存リスクへの配慮

- AIとの会話に時間制限・回数制限を設ける（1日に長時間は促さない）
- 会話が一定時間続いた場合、「今日はここまでにしておきましょうか」と自然に提案
- ロス後の特定期間（死後1週間以内）はAI会話機能を積極的には提示しない
- 定期的に「リアルの人との時間」「外の空気」を気持ちよく促す

### 13.6 データプライバシー設計

- 写真・記録データはユーザーのデバイスおよびサーバーに暗号化保存
- AI処理のためにデータを外部モデルに送信する場合は明示・同意取得
- アカウント削除時、すべてのデータを完全削除（エクスポート機能を先に提供）
- プライバシーポリシーを平易な日本語で提供

## 14. gstack活用前提のUX品質設計

### 14.1 初見ユーザーが迷わない導線設計

gstackでのブラウザテストで最初に確認すべきは「何のアプリかが30秒でわかるか」である。

**設計チェックポイント:**
- アプリ名とタグラインが最初の画面で伝わるか
- 「ペット登録」が最初のアクションとして自然に示されているか
- 登録フォームの項目数は最小化されているか（必須は3項目以内）
- 最初の記録追加までの操作が3タップ以内か

**失敗パターン:**
- オンボーディングが長すぎる（3スライド以上はスキップされる）
- 最初の画面が「機能説明」になっている
- プロフィール登録フォームが長い（全部入れないとスキップできない）

### 14.2 感情的に重いテーマでも触りやすいUIか

**設計チェックポイント:**
- 色調が重すぎない（ダークグレー・黒を基調にしない）
- フォントサイズが読みやすい（本文14〜16sp以上）
- ボタンが大きく・タップしやすい（最小44×44dp）
- 「触ってみたい」と思える視覚的な魅力があるか
- 画面を開いたとき「ほっとする」感覚があるか

**gstackでの確認:**

- ホーム画面 → 情報量が多すぎないか・詰め込み感がないか

### 14.3 「記録したい」「残したい」と思える導線か

**設計チェックポイント:**
- 記録ボタンが常に目に見える場所にあるか（ホームのメインCTA）

- 過去の記録を見ることで「もっと残したい」動機が生まれるか
- タイムラインが美しく・スクロールしたくなるかどうか

### 14.4 「亡くなった後に使うアプリ」と感じさせすぎない設計

**設計チェックポイント:**
- ファーストビューにロスケア・死後・別れのワードが出ていないか
- コピーが「楽しさ・残す喜び」を前面に出しているか
- アプリのビジュアルが「明るく・温かく・生活感がある」か
- App Storeのスクリーンショット・説明文が「生前利用」を前提にしているか

### 14.5 世界観と使いやすさの両立確認

**評価項目:**
- 世界観のために操作性が犠牲になっていないか（余白多すぎてスクロールが長い等）
- アニメーションが「待たされる」感じになっていないか
- タップ領域が小さすぎないか（デザインのために機能を犠牲にしない）

## 17. プロレベルに仕上げるための品質基準

### 17.1 UI品質基準

**なぜ必要か:** UIの品質はユーザーが最初に受け取る「信頼」である。特にセンシティブな感情を扱うアプリにおいて、粗いUIは信頼を壊し、記録という個人的な行為への参加を阻む。

| 基準 | 内容 |
|------|------|
| デザインの一貫性 | 全画面でフォント・色・余白・アイコンスタイルが統一されている |
| タップターゲット | 最小44×44dp、主要CTAは56dp以上 |
| 写真表示 | 元の比率を保持・高品質表示・読み込みスケルトン表示 |
| アニメーション | 200〜300msのイーズ、突然の画面切り替えがない |
| ダークモード | ライト/ダーク対応（特にダーク時に暗すぎない配色） |
| アクセシビリティ | コントラスト比4.5:1以上、フォントスケール対応 |

### 17.2 UX品質基準

**なぜ必要か:** UXの品質は「使い続けるかどうか」を決める。感情的に重い体験をするアプリで、操作に迷ったり不満を感じたりすると、ユーザーはアプリから距離を置く。

| 基準 | 内容 |
|------|------|
| 初回記録まで | オンボーディングから最初の記録保存まで5分以内 |
| タスク完了率 | 主要タスク（記録・予定追加）のエラーなし完了率90%以上 |
| レスポンス速度 | タップから反応まで100ms以内、画面遷移500ms以内 |
| エラーリカバリ | エラー発生時も「どうすればいいか」が明確に示される |
| 空状態設計 | 記録が0件のときのホームが「冷たく」ない |

### 17.3 文言品質基準

**なぜ必要か:** このアプリのコピーは、ユーザーの感情に直接触れる。一言の言葉が「続けたくなる」か「距離を置きたくなる」かを分ける。

| 基準 | 内容 |
|------|------|
| 一貫したトーン | 全コピーが「穏やか・温かい・押しつけない」トーン |
| 行動促進 | 命令形ではなく誘い形（「残す」「見てみる」） |
| NG表現の排除 | 断定的死後観・強制的前向き・医療代替の表現なし |
| 長さ | ボタンラベルは10文字以内、説明文は2行以内 |
| 見直し基準 | 全文言を「悲しみの中のユーザーが読んで傷つかないか」でレビュー |

### 17.4 感情設計品質基準

**なぜ必要か:** 機能が完璧でも、感情設計が失敗すると「使いたくない」アプリになる。ペットロスという文脈では、感情設計の失敗がユーザーを傷つける可能性がある。

| 基準 | 内容 |
|------|------|
| フェーズ感知 | アプリがユーザーのフェーズ（日常/不安/ロス直後/回復）を感知し、適切なUIを提供 |
| 安心感 | どの状態でアプリを開いても「責められない」感覚 |
| 選択の自由 | どの機能も強制されない・任意で使える設計 |
| 感情の証人 | AIがユーザーの感情を「評価しない」「否定しない」 |
| 適切な距離 | AIが近すぎず・遠すぎず。「一緒にいる」感覚 |

### 17.5 信頼性品質基準

**なぜ必要か:** 個人的な写真・感情記録・家族の記憶を預けるアプリである。データ消失・プライバシー侵害は、ユーザーへの重大な裏切りとなる。

| 基準 | 内容 |
|------|------|
| データ保全 | 記録のバックアップ・同期（クラウド保存必須） |
| 暗号化 | 転送時・保存時の暗号化（TLS 1.3 / AES-256） |
| プライバシー | AI処理時のデータ最小化・匿名化 |
| 削除の保証 | アカウント削除から30日以内の完全削除 |
| 障害通知 | サービス障害時のユーザーへの迅速な通知 |
| 可用性 | 99.9%以上のアップタイム目標 |

### 17.6 センシティブ領域としての安全性基準

**なぜ必要か:** ペットロスはメンタルヘルスに深刻な影響を与える。アプリが「悲しみを強化する装置」にならないよう、設計レベルで安全性を担保する必要がある。

| 基準 | 内容 |
|------|------|
| 依存防止 | AI会話の連続利用に自然な区切りがある |
| 専門家誘導 | 深刻な状態のシグナルを感知した場合、専門家リソースを提示 |
| 表現審査 | 全AI出力を事前に感情安全性レビューで審査 |
| テスト | ペットロス経験者によるユーザーテストを実施 |
| 更新審査 | 新機能リリース前にセンシティブ領域専門家のレビューを設ける |

## 18. 競争力整理

### 18.1 なぜこのアプリは埋もれないのか

**差別化ポイント1: 「生前から使う設計」が競合にない**

既存のペットロスアプリはロス後専用、ペット管理アプリは生前専用。この両者の間に存在するポジション──「生前の日常記録がそのままロスケアになる」──は現時点で競合が存在しない。このポジションの確立が、市場での独自性を保証する。

**差別化ポイント2: AIが「個人の記録」に基づいて寄り添う**

汎用グリーフAIと異なり、このアプリのAIはユーザーが積み上げた固有の記録を読んでいる。「リクが好きだったもの」「一緒に行った場所」「あの日の気持ち」を踏まえた言葉は、どんな汎用的なコンテンツより強く心に届く。

**差別化ポイント3: 感情設計の徹底がブランドになる**

機能の差別化は模倣される。しかし「触ったときの感覚」「言葉の温度」「余白の質」は、徹底した感情設計の積み重ねによってのみ生まれる。この感情的な質が、ユーザーの「このアプリだから」という選択理由になる。

### 18.2 なぜユーザーが継続したくなるのか

- **「1年前の今日」通知:** ユーザーが意識していないタイミングで「残しておいてよかった」と感じさせる
- **記録が資産になる:** 記録が増えるほど、タイムラインが豊かになる。やめると記録が途切れる感覚が継続動機になる
- **AIとの関係:** 記録が増えるほどAIがより「あなたの話」ができるようになる。蓄積に比例して価値が上がる設計

### 18.3 なぜ他人に勧めたくなるのか

- ペットを飼っている友人に「こんないいアプリある」と言いたくなる体験設計
- 写真・タイムラインをシェアする動機（SNS連携）
- 「ペットを亡くした人を支えたい」という動機（ロスケア文脈での紹介）
- アプリ自体のビジュアル品質が高く「人に見せたい」UIになっている

### 18.4 なぜ"普通のペットアプリ"より強いのか

普通のペットアプリは「管理」を目的に使われる。管理が終われば動機がなくなる。このアプリは「愛情の積み重ね」を目的に使われる。愛情は終わらない。ペットが元気なときも、病気のときも、亡くなった後も、ユーザーがアプリを開く理由が「愛情」に基づいている。この違いが、継続率・エンゲージメント・課金率のすべてで差を生む。

### 18.5 どこで感情価値の差が出るのか

| 体験ポイント | 感情価値が出るタイミング |
|-------------|----------------------|
| 最初の記録保存 | 「残せた」という小さな達成感 |
| タイムライン閲覧 | 「こんなに一緒にいた」という豊かさの確認 |
| 「1年前の今日」 | 「残しておいてよかった」という後付けの正解感 |
| AIメッセージ | 「わかってもらえた」という受容感 |
| ロス後のタイムライン | 「あの子はここにいた」という存在の確認 |
| 手紙生成 | 「ずっと言えなかったことが言えた」という解放感 |

### 19.2 最初に作りすぎないために削るべきもの

| 削るもの | 理由 |
|---------|------|
| 健康記録・体重グラフ | ペット管理アプリとの混同リスク。差別化を守る |
| コミュニティ機能 | モデレーション負荷が高い。品質確保困難 |

### 19.3 この企画の差別化ポイント（最重要3つ）

**1. 「生前から使う」唯一のグリーフ設計**  
生前の日常利用がそのままロスケアになる体験の流れは、業界に存在しない。ここが参入障壁であり、コアの競争優位。

**2. 個人の記録に基づくAI寄り添い**  
汎用的な慰めではなく、「あなたとあなたのペットの話」ができるAI。パーソナルな文脈が感情的な深さを生む。

**3. 感情価値で差別化されたUI/UX品質**  
機能ではなく「触った感覚」「言葉の温度」「体験の美しさ」で選ばれるアプリ。この品質の追求が模倣困難な参入障壁になる。

### 19.4 gstackを使った初回レビュー手順案

- gstackをセットアップし、スクリーンショット自動取得を設定

#### Step 2: 初回体験フロー検証
```

   確認: 3スライドで世界観が伝わるか
2. ペット登録 → 必須項目のみで完了できるか
   確認: フォームの心理的ハードル
3. 最初の記録追加 → 写真1枚+タグ1つで完了
   確認: 3タップ以内か・完了感があるか
4. タイムライン確認 → 1件でも「美しく見えるか」
   確認: 空状態 vs. 1件状態の差
```

#### Step 3: 日常利用フロー検証
```
5. 予定追加 → 通院予定を入力
   確認: カテゴリ選択の迷いがないか
6. ホーム → 予定・記録のバランスが良いか
   確認: 情報過多になっていないか
7. タイムラインスクロール → 10件以上の記録でのUX
   確認: スクロールの気持ちよさ・写真の見え方
```

#### Step 4: 感情品質確認
```
8. 全画面のコピーをリストアップ → NG表現チェック
9. 色・フォント・余白の一貫性確認
10. タップ時のフィードバック確認（ハプティクス等）
```

#### Step 5: 記録・分析
- 各Stepのスクリーンショットを保存
- 迷ったポイント・離脱ポイントをIssueに起票
- 「感情的に重かった/軽すぎた」瞬間を記録

## Appendix: 仕様書セルフレビュー

本仕様書を以下の観点で厳格にレビューした結果を記録する。

### レビュー結果

**Q1: 普通のペット管理アプリに落ちていないか**  
→ **問題なし。** 健康記録・体重グラフをMVPから意図的に削除し、「管理」ではなく「愛情の積み重ね」にフォーカスした設計になっている。タグラインも「記録じゃなく、愛情を残す。」と明示。

**Q2: 感情価値が弱くないか**  
→ **問題なし。** フェーズ設計・AIの設計原則・コピーライティング原則・感情設計品質基準を通じて、感情価値を設計の核心に置いた。「1年前の今日」通知・手紙生成・AI寄り添いにより、機能ベースではなく感情ベースの差別化を明示した。

**Q3: MVPが広すぎないか**  
→ **問題なし。** MVP必須機能を5つに絞り（プロフィール・記録・予定・タイムライン・通知）、AI機能・家族共有・手紙生成をPhase 2以降に明確に分離した。「削るべきもの」セクションで意図を明示した。

**Q4: 生前利用とロスケアの接続が自然か**  
→ **問題なし。** 「生前の日常記録がそのままロスケアになる」という接続を、フェーズ設計・シナリオ5・競争力整理で繰り返し設計している。接続点はステータス切り替えUIと、タイムラインの継続閲覧で担保される。

**Q5: センシティブ領域として危うい表現がないか**  
→ **問題なし。** 「虹の橋で会える」「天国で待っている」等の断定的表現を明示的にNGとし、代替方針を設けた。「乗り越える」「立ち直る」「前を向いて」もNGに含めた。

**Q6: gstackで実地検証したくなるレベルまで具体化できているか**  
→ **問題なし。** シナリオ6件・確認観点・初回レビュー手順（Step 1〜5）を詳細に記述した。gstackでの確認ポイントを各シナリオに明示した。

**Q7: プロレベルのアプリ品質を目指す内容になっているか**  
→ **問題なし。** UI/UX/文言/感情設計/信頼性/安全性の6つの品質基準を設け、それぞれ「なぜ必要か」から記述した。単なるチェックリストではなく、判断基準として機能する内容にした。
