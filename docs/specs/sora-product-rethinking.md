## 1. 経営者目線の総評

### 1.1 現状への率直な評価

**コンセプトは本物だが、今日この瞬間に使えるアプリになっていない。**

既存仕様の「生前の日常記録がそのままロスケアになる」というポジションは本質的に正しく、競合に存在しない。しかし現状のアプリには致命的な問題が重なっており、このままリリースしてもユーザーはDay 1で離脱する。

**致命的問題（今すぐ解決が必要）:**

1. **データが消える（ISSUE-001: critical）**  
   リロードで全記録が消える。これはアプリとして機能していない。ペットとの大切な写真・記録が消えるアプリは、どんなに美しくても使われない。信頼がゼロになる。

2. **アプリタイトルが「ペットロスヒーリング」（ISSUE-006: critical）**  
   これが最大の致命傷。「ペットロスヒーリング」を名乗った瞬間に、健康なペットと暮らすユーザーの90%が離脱する。アプリのコアユーザーを取り逃がしている。

3. **プロフィール登録のコピーが過去形（ISSUE-007: critical）**  
   「〜でした」「〜していました」という過去形は、生きているペットを持つユーザーに「このアプリは亡くなった後に使うものだ」という強烈なシグナルを送る。新規ユーザーの入口を自分で壊している。

4. **気持ち記録の選択肢がロス後専用（ISSUE-008: high）**  
   「悲しい」「寂しい」しかない感情タグは、幸せなペットライフを送るユーザーには使えない。生前ユーザーの記録習慣形成を根本から妨げている。

**構造的な弱点:**

- **「毎日開く理由」が初年度に存在しない**: 「1年前の今日」機能は使い始めて1年経たないと価値が出ない。では最初の365日、ユーザーが毎日アプリを開く動機は何か？ 現状の答えは「記録したいから」だけであり、これは習慣として弱い。
- **幸せの可視化が設計されていない**: 「どれだけ幸せを共有したかを残す」という強い価値観があるのに、それを実際に見える形で提示する設計が存在しない。
- **新しく迎えたユーザーのオンボーディングが別設計になっていない**: 「子犬を今日迎えました」「子猫が来ました」という最高の導入タイミングを設計できていない。

### 1.2 強みの確認

**本当に強い点は以下3つだけ持ち続けること:**

1. **「生前から使う唯一のグリーフ設計」** — これを失ったら存在意義がなくなる
2. **「個人の記録に基づくAI寄り添い」** — 汎用グリーフAIには絶対に出せない体験
3. **「感情価値で差別化された美しいUI」** — 機能競争から降りる戦略として正しい

### 1.3 総評まとめ

コンセプトの方向性は正しい。しかし実装とコピーが「ロス後専用アプリ」になっており、日常利用ユーザーを取り込めていない。MVPの前に、根本的なポジション修正が必要。

## 2. プロダクト再定義

### 2.1 ポジションの明確化

**今の問題:** コンセプトでは「生前から使える」と言いながら、実装レベルではロス後専用になっている。

**再定義:**

> **Soraは「ペットとの幸せを積み重ねるアプリ」であり、「ペットロスアプリ」ではない。**

「グリーフケア」という言葉を表に出すことをやめる。Soraが提供するのは「今日という幸せな1日を、未来の自分への贈り物として残す体験」である。

### 2.2 コアループの再設計

**現状のコアループ（弱い）:**
```
記録する → タイムラインを見る → また記録する
```

**再設計後のコアループ（強い）:**
```
今日の思い出を残す（30秒）
    ↓
「一緒に過ごした幸せ」が蓄積されていく（可視化）
    ↓
「今日で〇〇日一緒にいたね」「今月の思い出 X枚」が見える
    ↓
「もっと残したい」という感情が生まれる
    ↓
続けるほど価値が上がる
    ↓
「1年前の今日」が届く → 「残しておいてよかった」
    ↓
離れられなくなる
```

このループのキーは**蓄積の可視化**と**継続の報酬設計**。記録するだけでなく、蓄積が「見える」ことが継続動機になる。

### 2.3 タグラインの再考

**推奨タグライン:**

> **「一緒にいる今が、ずっと残る。」**

理由:
- 「今」を強調することで生前ユーザーへの訴求になる
- 「ずっと残る」はロスへの備えを示唆するが押しつけない
- 短く、温かく、ロスを想起させすぎない

**NG:**
- 「そばにいた時間が、ずっとそばにいてくれる。」→ 過去形スタートでロス後を想起させる
- 「記録じゃなく、愛情を残す。」→ 抽象的すぎてアプリの実態が伝わらない

### 2.4 アプリの見え方の再設計

**App Store / SNSでの見え方:**

- スクリーンショット1: 「愛犬の今日を、一生残す。」（写真投稿画面）
- スクリーンショット2: 「一緒に過ごした〇〇日間。」（幸せスコア画面）
- スクリーンショット3: 「1年前の今日のリク。」（振り返り通知）
- スクリーンショット4: 「あなたとリクの物語。」（タイムライン）
- スクリーンショット5: 「今日の気持ちを残す。」（気持ち記録）

**掲載説明文:**  
「ペットと一緒に過ごす毎日を、美しく・気軽に・ずっと残せるアプリです。写真一枚から始まる記録が、やがて『あなたとこの子だけの物語』になります。」

## 3. ターゲット整理

### 3.1 ターゲットセグメント一覧

```
優先度: A > B > C > D > E
```

### A. メインターゲット: 新しくペットを迎えた人（20〜35歳）

**なぜメインか:**  
これが最高の導入タイミングだからだ。「子犬を迎えた初日から全部記録したい」という感情は、記録習慣の形成に最も強い。1年・3年・5年分の記録が蓄積されれば、離脱コストが極めて高くなる。

| 項目 | 内容 |
|------|------|
| 課題 | 成長の記録が断片化している。Instagramに投稿するが体系化されない。いつどこで何したか後から分からなくなる |
| 感情 | 「全部残したい」「成長が嬉しい」「見せたい」「後悔したくない」 |
| 使う理由 | 子犬・子猫の成長記録を1か所にまとめたい |
| 継続する理由 | 続けるほど「うちの子の物語」が充実する。月齢の変化が見える |
| 必要な機能 | 写真投稿、気持ち記録、「今日で〇〇日目」表示、月ごとの成長まとめ |

**獲得戦略:** ペットショップ・ブリーダー・保護犬団体との提携。「迎えた日から始めるアプリ」として認知させる。

### B. 日常ユーザー: ペットと暮らす25〜45歳の人（主に女性）

**なぜ重要か:**  
既存仕様の想定ターゲット。感情記録・課金意欲・SNS共有行動のすべてで最も高いポテンシャルを持つ。

| 項目 | 内容 |
|------|------|
| 課題 | スマホに写真は溢れているが、見返せていない。感情が記録に残っていない |
| 感情 | 「残したいが面倒」「記録は好きだが続かない」「思い出をもっと大切にしたい」 |
| 使う理由 | 今日の小さな幸せを30秒で残せる。ペットとの記念日を忘れない |
| 継続する理由 | 「1年前の今日」通知。蓄積した記録の美しさ。習慣化による離脱コスト |
| 必要な機能 | 写真投稿、気持ちタグ、タイムライン、「今日で〇〇日」、記念日通知、幸せスコア |

### C. 高齢・闘病ペットの飼い主（35〜65歳）

**なぜ重要か:**  
課金意欲が最も高く、記録意欲も高い。「今のうちに残したい」という切実な動機がある。このセグメントへのケアがSoraの独自価値に直結する。

| 項目 | 内容 |
|------|------|
| 課題 | 「もっと早くから記録しておけばよかった」という後悔への恐れ。通院・投薬管理が煩雑 |
| 感情 | 焦り・愛情の高まり・「今できることをしたい」 |
| 使う理由 | 残り時間を意識した記録。通院記録と感情記録を一元化したい |
| 継続する理由 | 記録が増えるほど「後悔なく愛した」という確信が深まる |
| 必要な機能 | 写真・メモ、通院記録、投薬記録、気持ちタグ、「今日で〇〇日一緒にいる」 |

**設計上の注意:** 「いつかのために」という表現は絶対に使わない。あくまで「今日を残す」というフレーミング。

### D. ペットロス直後のユーザー（30〜60代）

**なぜ重要か:**  
既存仕様のコンセプト上の主役。ただしこのセグメントを「入口」にするのは間違いで、「生前からの利用者がここに至る」設計が正しい。新規ユーザーとしての流入は限定的。

| 項目 | 内容 |
|------|------|
| 課題 | 過去の記録を整理・見返したいが、ツールがない。孤独に悲しみを抱えている |
| 感情 | 混乱・空白・「いなくなった現実を受け入れられない」 |
| 使う理由 | 蓄積した記録に静かにアクセスしたい |
| 継続する理由 | 「あの子はここにいた」という存在の確認。アプリが「安全な場所」になる |
| 必要な機能 | タイムライン閲覧（新規投稿不要）、気持ち記録、静かな存在感、AI振り返り（Phase 2） |

**注意:** ロス直後ユーザーへの新規流入を積極的に獲得するマーケティングはしない。コンテンツ・設計は「生前から使っている人がロスを経験した場合」を前提にする。

### E. これからペットを迎える人

**なぜ重要か:**  
「迎えた日から記録する」という行動は最高の出発点。ただしアプリがない段階での訴求が難しい。App Store経由の自然流入を想定。

| 項目 | 内容 |
|------|------|
| 課題 | 迎える前からどんなアプリを使うか決めていない |
| 感情 | 期待・準備したい気持ち・「全部残したい」 |
| 使う理由 | 迎えた初日から始めるための準備 |
| 継続する理由 | 最初から使い始めているため習慣が形成されやすい |
| 必要な機能 | 初日登録のUX設計。ペット名・誕生日だけで始められる |

## 4. 提供価値

### 4.1 機能価値（使える）

- **記録の一元化:** 写真・メモ・気持ち・予定がひとつにまとまる
- **時系列の可視化:** 何年何月に何があったかが美しく見える
- **予定の管理:** 通院・トリミング・ワクチンの抜け漏れをなくす
- **継続の可視化:** 「一緒に過ごした日数」「今月の記録数」「共有した幸せの総量」が数字で見える

### 4.2 感情価値（使いたくなる）

- **「残してよかった」の後付け充足感:** 「1年前の今日」通知が届くたびに、記録した過去の自分に感謝する
- **「こんなに愛していた」の自己確認:** 積み上げた記録が「あなたはこれだけ愛していた」という証になる
- **「今日の小さな幸せが永遠になる」感覚:** 特別な日でなくても、残す価値があると思える
- **ロス後の「ひとりじゃない」安心感:** 記録がそのまま癒しになる

### 4.3 社会価値（人に話したくなる）

- **「うちの子の物語」を友人に見せたい:** SNS的な見せたい感情を満たしながら重くならない
- **「ペットロスを理解してほしい」という欲求への応答:** 記録が「私はこれだけ愛していた」という社会的な証になる
- **ペットとの別れを、一人で抱えない体験:** 日本社会でまだ不十分なペットロスへの理解を、個人レベルで補う

## 5. 競合との差別化

### 5.1 競合マップ

| 競合 | 強み | 弱み | Soraとの差 |
|------|------|------|-----------|
| PetNote / 動物病院アプリ | 健康管理・医療記録の精度 | 感情ゼロ・管理ツール感 | 感情記録・愛情の可視化 |
| Instagram / SNS | 写真の見栄え・共有 | 体系化されない・振り返れない | 時系列の蓄積・ペット専用 |
| 日記アプリ（Day One等） | 継続記録・バックアップ | ペット特化なし・感情設計なし | ペットとの文脈・気持ちタグ |
| グリーフアプリ（国内外） | ロス後ケアの深さ | 生前から使えない | 生前から始まる自然な流れ |
| AI会話アプリ | 会話の深さ | 記録との文脈連携なし | 「あなたとあなたのペットの話」 |

### 5.2 Soraにしかできないこと（3つに絞る）

**①「積み重ねることで価値が増す」設計**  
記録が増えるほどタイムラインが豊かになり、AIがより「あなたの話」ができるようになる。他のアプリではこの「蓄積に比例した価値増加」が設計されていない。

**②「日常のままロスケアになる」シームレスな移行**  
生前の記録が、一切の変更なくロス後の癒しになる。これは設計思想であり、機能リストではない。競合が模倣しようとしても、過去のデータがなければ実現できない参入障壁。

**③「幸せの総量を可視化する」体験**  
「一緒に過ごした〇〇日」「笑った〇〇回」「お出かけ〇〇回」。ペットとの幸せを数字と文脈で見える形にする。これは記録アプリでも、グリーフアプリでも提供できていない独自体験。

### 5.3 埋もれないためのポジション戦略

**App Storeカテゴリ:** 「ライフスタイル」に配置（「メディカル」「グリーフ」は避ける）

**キーワード戦略:**
- 「ペット日記」「犬 記録」「猫 記録」「ペット 思い出」をメインに
- 「ペットロス」「グリーフ」は避ける（流入後の価値として提供する）

**口コミ設計:**
- 「このアプリ、ペット飼ってる友達に絶対すすめたい」という紹介動機を設計
- タイムラインのスクリーンショット共有機能で、自然なUGCを生む

### 6.3 「毎日開く理由」の設計

**初年度（1年前の今日が存在しない期間）に毎日開く動機:**

1. **今日で〇〇日目の表示** — 「今日でモモとの720日目」が毎日変わる。見たくなる。
2. **今日の一言日記** — Twitterより短い、5秒で完了する。習慣コストが低い。
3. **今週の思い出サマリー（月曜通知）** — 「今週〇枚記録しました」のフィードバック。
4. **記念日・誕生日リマインダー** — 「もうすぐリクのお誕生日です」でアプリを開く。
5. **予定後の記録誘導** — 「今日のトリミング、どうでしたか？」で記録の動機を作る。

## 7. キラー機能案

### 7.1 「幸せの総量」を見える化する機能

**機能名:** Sora Story（ソラストーリー）

**概念:**  
「あなたとリクが一緒に作ってきた物語の密度」を可視化する。単なる統計ではなく、感情的な物語として表示する。

**構成要素:**

```
📅 一緒に過ごした日数: 847日
📸 残した思い出: 283枚
🏥 一緒に通った病院: 12回
✂️ トリミング記録: 8回
🌙 今日の気持ち記録: 156回
🎉 記念日: 4回
🐾 お出かけ記録: 23回
```

**感情的な表示（これが差別化）:**  
単なる数字でなく、文脈として表示する。

> 「あなたとリクは、847日間一緒にいました。  
>  283枚の写真を残し、12回一緒に病院へ行きました。  
>  156日間、今日の気持ちを記録しました。  
>  これがあなたとリクの物語です。」

**なぜキラーか:**
- 毎日数字が変わる → 毎日開く理由
- 「こんなに一緒にいたんだ」という発見 → 感情的な価値
- ロス後に見返したとき → 「後悔なく愛した」という確信に変わる
- SNSシェアしたくなる → UGC・バイラル

### 7.2 「今月のリク」自動まとめ

**機能名:** Monthly Story（マンスリーストーリー）

毎月1日に、先月の記録から自動生成される「今月の〇〇」。

**内容例（AI生成 + テンプレート）:**

> 「2026年3月のリク」  
> 今月は12枚の写真を残しました。  
> 最もよく記録された気持ちは「愛おしい」（8回）。  
> トリミングに行き、3回お散歩の記録がありました。  
> 今月一番の思い出: [最もいいね数/保存数の写真]

**なぜキラーか:**
- 毎月届く → 継続の動機
- 振り返りの達成感 → 記録習慣の強化
- 月ごとに「リクの成長・変化」が見える
- フォトブック出力への自然な誘導（Phase 3）

### 7.3 「去年の今日」通知（既存機能の再定義）

**現状の問題:** 「1年前の今日」という名前が過去感を強調しすぎる。

**再定義:**

> 「リクとの思い出が届きました 🐾」  
> [サムネイル]  
> 「1年前の今日、こんなことがありました。」

**なぜ強力か:**
- 記録した価値の「後付け証明」になる
- 「残しておいてよかった」という感情が記録習慣を強化する
- フィードバックループが完成する

### 7.4 「今日で〇〇日目」カウンター

**ホーム画面の中心に配置する数字:**

```
🐾 リクと一緒に
   847 日
```

この数字が毎日増える。シンプルだが強い。

**なぜ強力か:**
- 開くたびに数字が変わる（変化の演出）
- 「もうすぐ1000日」というマイルストーン意識が生まれる
- ロス後も「一緒にいた日数」として永遠に残る

### 8.1 MVPの目的

## 11. 収益化案

### 11.1 基本方針

**原則: 感情の深さと課金の深さを連動させる**

感情価値が高い機能（AI・手紙・アルバム）を課金に結びつける。  
日常的な記録（写真・タイムライン）は無料。これが習慣形成の入口になる。  
「課金しないと使えない」ではなく「課金したくなる」設計。

**感情的に問題のある課金設計（絶対避ける）:**
- ロス後のユーザーへのアップセル通知
- 「もっと思い出を残すためにプレミアムに」という促し
- 写真容量の上限で記録を止めること（無料の基本容量を十分に確保する）

### 11.2 無料で提供すべき範囲

| 機能 | 理由 |
|------|------|
| ペット登録（1頭） | 入口であり、広告媒体 |
| 写真記録（月100枚まで） | 習慣形成に必要 |
| タイムライン閲覧 | 価値体験の中心 |
| 「今日で〇〇日目」 | 毎日開く動機。課金に閉じてはいけない |
| 気持ちタグ記録 | 記録習慣の核心 |
| 予定管理（基本） | 日常利用の維持 |
| 「去年の今日」通知（基本） | ロック後の継続動機 |

### 11.3 サブスク向き機能（月額 980円）

**プレミアムプランの課金ポイント:**

| 機能 | 課金理由の強さ |
|------|--------------|
| ペット登録（無制限） | 2頭以上飼っている人には必須 | ★★★ |
| 写真容量無制限 | 「大切な記録を削除したくない」という心理 | ★★★ |

- 理由: 480円は安すぎて「たかがペットアプリ」というシグナルを送る。ペットへの課金意欲は高く、980円は感情価値に見合う。

## 12. 倫理設計

### 12.1 やってはいけない表現

| カテゴリ | NG表現 | 代替表現 |
|---------|--------|---------|
| 断定的死後観 | 「天国で待っている」「虹の橋で会える」「また会える」 | 「多くの人がそう感じています」「あなたはそう思ってよい」 |
| 宗教的断定 | 特定宗教の死後観・浄土・天国 | 「ここにいた証として」「記録が続く」 |
| 強制的前向き | 「前を向いて」「乗り越えて」「新しいペットを」「立ち直る」 | 触れない。ユーザーから言わない限り言わない |
| 医療代替示唆 | 「癒されます」「心が治ります」「安心できます（断言）」 | 「そばにいます」「一緒にいます」 |
| 依存を促す | 「毎日必ず開いてください」「Soraだけが支えです」 | 「気が向いたときに」「あなたのペースで」 |
| 数値的プレッシャー | 「今月〇件しか記録していません」「記録が少ないです」 | 言わない。記録しないことを責めない |

### 12.2 AIが踏み込みすぎてはいけない領域

- **診断・予後の言及:** 「リクは〇〇病だったのでしょう」「回復するはず」等
- **感情の評価・正誤判断:** 「その感情は正しい/正しくない」
- **別れへの心理的準備の促し:** 「もうすぐお別れかもしれないですね」等
- **ペットの擬似人格化:** 「リクはきっとこう思っています」「リクからのメッセージ」として語る
- **長期的な感情処方:** 「毎日開いて私と話しましょう」「Soraがいれば大丈夫」

### 12.3 「亡くなった子本人」と誤認させない設計

**手紙機能（Phase 3）の設計方針:**

- 手紙の主体は「あなたがリクに伝えたかった言葉」であり「リクからの手紙」ではない
- UIで明示: 「これはあなたの記録から、あなたの言葉で生成した手紙です」
- AIが「リク目線」で語ることは禁止。主体は常にユーザーの感情
- 生成前に「今の気持ちで大丈夫ですか？ 少し落ち着いてから読みますか？」という確認UI

### 12.4 依存を強めすぎない設計

- AI会話が1日30分以上になった場合、「今日はここまでにしませんか？外の空気を吸うのもいいかもしれません」と自然に提案
- AI会話機能はロス後7日以内はデフォルトOFF（ユーザーが選択すれば使える）
- アプリ内に「専門家のサポートについて」ページを設ける（設定の中に静かに配置）

### 12.5 医療・カウンセリングとの線引き

アプリ内テキスト（設定画面）:
> 「Soraは、ペットとの思い出を大切にするためのアプリです。深刻な悲しみや日常生活への影響が続く場合は、専門家のサポートをご検討ください。」

**リンク先（静かに配置）:**
- 一般社団法人ペットロス・コンパニオンアニマル研究所
- 心の健康相談統一ダイヤル（厚生労働省）

### 12.6 緊急時・深刻なメンタル不調への対応

AIが以下のシグナルを検出した場合:
- 自傷・自殺に関連する言葉
- 極度の絶望・無力感の繰り返し

対応:
1. AIの応答を一時停止
2. 「今、つらい状況にあるようです。あなたのことが心配です。」と伝える
3. 専門家のサポートへのリンクを提示
4. 通常の会話機能への自動復帰はしない

### 12.7 優しいが、断定しない文言ルール

**良い文言:**
- 「〇〇だったのかもしれませんね」
- 「そう感じていいと思います」
- 「あなたのペースで大丈夫です」
- 「今日の気持ちを、残しておきましょうか」

**NG文言:**
- 「〇〇でした（断定）」
- 「きっと大丈夫です」
- 「時間が解決します」
- 「強くなりましょう」

- プレミアム誘導がある場合、適切なタイミングか

### シナリオ7: ペットロスを知らないユーザーが「この子に何かあったら使えるかも」と感じる体験

**前提:** 日常ユーザーがタイムラインを見ながら「もし…」と考えるとき。

**検証観点:**
- アプリがロスケアの側面を「存在するが前面に出さない」設計になっているか
- ロスケア機能への導線が「必要な時に見つけられる」程度に静かに配置されているか
- 「ロスに備えている」感じがなく、「今を大切にしている」感じがあるか

### シナリオ8: 色・文言・余白がSoraブランドに合っているか（総合UX確認）

**確認ポイント:**
- 全画面でフォント・色・余白の一貫性
- 暖色系（クリームホワイト・アンバー・セージグリーン）の統一
- コピーが全て「穏やか・温かい・押しつけない」トーン
- ロードスクリーン・空状態・エラー画面でも冷たくないか

## 17. リスクと対策

### 17.1 プロダクトリスク

| リスク | 深刻度 | 対策 |
|--------|--------|------|
| 「ペットロスアプリ」として認識されて日常ユーザーが離脱 | 🔴 高 | App Store掲載文・スクリーンショットの徹底した生前利用訴求。「ペットロス」「グリーフ」をキーワードから外す |
| 記録習慣が形成されず初月離脱 | 🔴 高 | 「今日で〇〇日目」カウンター・Weekly通知・記録のフリクションゼロ設計で習慣コストを最小化 |
| データ消失・クラウド障害 | 🔴 高 | Supabase冗長化・定期バックアップ・エクスポート機能の早期提供 |
| AI生成コンテンツが感情的に不適切 | 🟠 中 | Phase 2前に全プロンプトの感情安全性テスト。ペットロス経験者によるユーザーテスト必須 |
| ロス後ユーザーへのAI過依存 | 🟠 中 | 利用時間制限・専門家リソース案内・会話の自然な区切り設計 |

### 17.2 事業リスク

| リスク | 深刻度 | 対策 |
|--------|--------|------|
| 課金率が低く収益化できない | 🟠 中 | 「写真無制限」「家族共有」という実用的価値でのサブスク訴求。感情的な課金より実用的な価値で設計 |
| 競合（大手ペット管理アプリ）が同機能を追加 | 🟡 低〜中 | 感情設計の質とAIの個人文脈が模倣困難。早期のユーザー獲得と記録蓄積が参入障壁になる |
| ペットロス領域の倫理問題で炎上 | 🟠 中 | 倫理設計の徹底・ペットロス専門家の監修・透明なAI利用方針の公開 |
| エンジニアリングリソース不足でMVPが遅延 | 🟡 中 | 機能を絞り込む。AI機能はPhase 2以降。MVPの8週間スコープを厳守 |

### 17.3 感情設計リスク（最重要）

| リスク | 深刻度 | 対策 |
|--------|--------|------|
| アプリを開くたびに悲しみが増す体験 | 🔴 高 | ロス後ユーザーのタイムライン表示はソフトなフィルタリングを提供。「今日は休んでもいい」メッセージ |
| AI会話が「ペットが生きているかのような」体験を作る | 🔴 高 | AIの主語は常に「あなたの感情」。ペット視点での語りは禁止 |
| 「去年の今日」通知がロス後ユーザーを突然傷つける | 🟠 中 | ペットステータスに応じて通知の文体・配慮を変える。ロス後は事前警告UIを表示 |
